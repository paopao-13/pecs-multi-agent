## 常见追问

### Q1:GAIA 92% 这个数字可信吗?官方榜单多少?

**答**:得拆开看。92% 是我自建的 28 题子集,不是官方 165 题。10 题里 5 道大数计算是启发式 0-token 秒杀的,算"免费得分";**真正体现框架推理能力的是 tokens>10 的推理题子集,准确率 92.3%(12/13)**——这部分是 Planner 拆解 + Critic 交叉验证 + Synthesizer 综合的完整四角色流程跑出来的,真实有效。

官方 165 题我没接上,因为 hf.co 接受 GAIA license 这步被网络拦了。预判官方分布下准确率 40-55%——官方 L1 计算题占比远低于我的子集,启发式优势发挥不出来。但代码骨架 `datasets/gaia_official_dataset.py` 已经写好,解决网络就能跑。

**关键点**:我不把 92% 说成"GAIA benchmark 92%",而是明确说"自建 28 题推理子集 92%"。诚实标口径比吹数字重要,招聘方懂行的会追问分布,露馅反而减分。

### Q2:Token -39% 是怎么算出来的?调度机制真的能省这么多?

**答**:-39% 是**端到端对比**——PECS 平均 442 tokens/task vs 同模型 ReAct 722 tokens/task。但这里有个对比假象我得说清楚:ReAct 在大数计算上失败后会重算,消耗本来就高,所以 39% 里有一部分是"ReAct 失控"带来的。

**纯预算调度机制本身的贡献只有 -4.5%**——这是消融实验测的,禁用启发式、只开关预算感知调度,紧预算 877 tok vs 宽预算 918 tok。剩下的省的是启发式路由(计算类任务零 LLM 调用,-99.4%)和失败重算减少。

**关键点**:报告里我区分了"机制贡献 -4.5%"和"端到端 -38.78%"两个口径。-39% 是真实测的、对比公平(同模型同工具同题),但不是调度单独的功劳。

### Q3:WebShop +18pp 达标了吗?

**答**:**没真达标,我得诚实说**。现在的 +100pp 是本地 8 商品 mock 适配器跑出来的,ReAct 0% 是 strawman 基线(没调 webshop 工具,LLM 直接编造答案)。这个数字不计入 +18pp 目标。

真实 AgentBench WebShop 环境我没跑通——它依赖 pyserini/Java,Windows 上极脆弱,沙箱又拉不到源码。但我做了完整的真环境脚手架:用 rank_bm25 纯 Python 替代 pyserini(接口对齐),写了 HTTP 桥把 gym 环境包成 /reset /step 端点,数据从 HF 镜像下绕开 Google Drive 被墙。按 `docs/webshop_local_runbook.md` 本机能跑,只差我没真去执行验证。

**关键点**:"接真实环境受限 → 诚实降级 + 写可复用脚手架"这个故事本身比硬凑 +18pp 可信。招聘方要 AgentBench 经验,我展示的是集成能力和工程权衡,不是吹数字。

### Q4:四角色分工到底比单 Agent 强在哪?

**答**:三个真实差异,都有数据支撑:

1. **大数计算不翻车**:ReAct 的 LLM 心算 2^30-2^20 这种会出错(算出 2^30 但忘记减 2^20),PECS 通过 Planner 识别计算任务 → 生成 Python 代码 → AST 沙箱精确执行。实测 ReAct 在 2 道大数减法题上全错,PECS 全对。
2. **错误不一路传递**:Critic 评审能拦截错误步骤,带反思信息打回 Planner 重规划。单 Agent 出错就错了,没人反馈。
3. **成本有上限**:Token 预算三级降级 + 角色独立配额,单任务成本可控。ReAct 反复调 LLM 直到完成,简单复杂任务成本差十倍。

代价是知识检索类任务 PECS Token 更高(+115%),因为四角色各调一次 LLM。但 Critic 的质量评审提供了 ReAct 没有的错误拦截——这是工程权衡,不是免费午餐。

### Q5:BM25 替代 pyserini 是偷工减料吗?

**答**:是工程权衡,不是偷工减料。pyserini 依赖 Java + Lucene + JNI,Windows 上极脆弱(部署过的人都知道是天坑),沙箱又没 Java。我读源码发现 `engine.py` 虽然 import 了 `rank_bm25` 但没接进搜索路径,只用了 Lucene。

所以我写了 `BM25SearchEngine` 实现相同的 `search(k)` / `doc().raw()` 接口,改 `init_search_engine` 默认返回 BM25,设 `WEBSITE_USE_PYSERINI=1` 才走原版 Lucene。**功能等价,部署门槛大降**(本机 conda py3.8 不用装 Java)。

代价是 BM25 与 Lucene 排序略有差异——BM25 是纯词频+文档长度归一化,Lucene 默认 BM25+字段权重。对 WebShop 这个场景(商品标题/描述检索)影响很小,但我作品集里如实标注了"搜索后端用 rank_bm25 纯 Python 实现以兼容 Windows,与原版 pyserini/Lucene 功能等价"。

### Q6:这个项目最难的部分是什么?

**答**:不是架构设计,是**让评测数字可信**。

架构搭出来一周就跑通了,但跑出来"100% 准确率、53 tokens/task"我反而不敢信——一查发现启发式把所有 LLM 调用都拦截了,数字是假的。修完启发式又发现 NameError 报错文本被当答案冻结,错误答案永远不重试。每个"看起来达标"的数字背后都可能藏着一个 bug 链。

所以这个项目我花时间最多的不是写功能,是**写消融实验、拆 Token 口径、诚实标注局限**。招聘方追问"这个 39% 怎么来的""WebShop 是真环境吗",我能拆开讲清楚机制贡献 -4.5% vs 端到端 -38.78%、本地 mock vs AgentBench——这种方法论诚实度,我觉得比数字多漂亮更重要。
