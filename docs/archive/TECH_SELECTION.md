# 技术选型决策报告

> 本文档记录 pecs-multi-agent 项目在开发前的技术选型分析过程，包括候选框架对比、决策理由和弃选原因。

## 1. 选型背景

项目核心需求：构建一个多智能体协作框架，实现 Token 预算感知调度，要求能够基于运行时状态动态路由和降级。因此框架必须满足三个硬性条件：

1. **显式状态管理**：共享状态在节点间传递，可追踪、可序列化
2. **条件路由**：基于状态动态决定下一步走哪个节点
3. **路由可编程**：降级逻辑能嵌入路由函数，而非外挂 if-else

## 2. 候选框架对比

| 维度 | LangGraph | AutoGen | CrewAI | 自研 |
|------|-----------|---------|--------|------|
| **状态管理** | StateGraph 显式管理，类型安全 | 对话历史隐式传递，无结构化状态 | 任务队列+上下文，半结构化 | 完全自定义 |
| **路由控制** | 条件边（conditional edges），细粒度 | 自由对话模式，轮次不可控 | 固定流程 sequential/pipeline | 完全自定义 |
| **成本控制** | 路由函数可嵌入预算检查 | 无内置机制，需外挂 | 无内置机制，需外挂 | 完全自定义 |
| **生态成熟度** | LangChain 生态，文档完善 | 独立生态，微软维护 | 独立生态，社区驱动 | 无生态 |
| **学习成本** | 中等（需理解 StateGraph） | 低（对话式自然） | 低（角色+任务直观） | 高（全部自研） |
| **可扩展性** | 节点/边可自定义 | Agent 可自定义 | Role/Task 可自定义 | 完全可控 |
| **调试支持** | 图可视化 + 状态追踪 | 对话日志 | 任务日志 | 需自建 |

## 3. 详细分析

### 3.1 LangGraph（最终选择）

**优势：**
- StateGraph 的条件边机制是唯一原生支持"基于状态动态路由"的框架
- 状态类型用 Pydantic BaseModel 定义，运行时类型校验
- LangChain 生态兼容，工具集和 LLM 封装可复用
- 图结构可视化，调试时能看到完整的节点流转路径

**劣势：**
- 学习成本高于 AutoGen/CrewAI，需理解 StateGraph、条件边、编译等概念
- 版本迭代较快（0.2.x），API 可能有 breaking changes
- 社区资源相对 AutoGen 少

**适配度评估：** 三项硬性条件全部满足。状态管理用 AgentState（Pydantic），条件路由用 `add_conditional_edges`，降级逻辑直接写在 `route_after_executor` / `route_after_critic` / `route_after_synthesizer` 三个路由函数中。

### 3.2 AutoGen（弃选）

**优势：** 多 Agent 自由对话模式，开发速度快，微软维护。

**弃选原因：**
- 对话轮次不可控：Agent 之间自由对话，无法预算化 Token 消耗
- 无条件路由：无法基于 Critic 评分决定"重试还是继续"
- 状态隐式传递：对话历史作为上下文传递，无法结构化管理 Token 预算、角色配额等字段
- 降级逻辑需外挂：无法嵌入框架路由，只能在每个 Agent 的 prompt 中加条件判断

**结论：** 适合快速原型和自由对话场景，不适合需要精确成本控制和条件路由的场景。

### 3.3 CrewAI（弃选）

**优势：** 角色（Role）+ 任务（Task）+ 流程（Process）三层抽象，直观易用。

**弃选原因：**
- 流程固定：sequential 和 pipeline 两种模式，无法基于运行时状态动态决定路由
- 无预算感知：Task 执行无 Token 预算概念，需外挂
- 路由不可编程：流程在定义时确定，运行时无法修改
- 缺乏自动纠错：无 Critic 角色，依赖人工 review

**结论：** 适合固定流程的多角色协作，不适合需要动态路由和自动纠错的场景。

### 3.4 自研框架（弃选）

**弃选原因：** 虽然灵活性最高，但开发周期长、测试成本高、缺乏社区支持。LangGraph 已提供足够的自定义能力（自定义节点、条件边、状态类型），无需从零构建图引擎。

## 4. LLM 选型

| 维度 | DeepSeek-V3 | GPT-4o | Claude 3.5 | Qwen-Max |
|------|-------------|--------|------------|----------|
| **API 价格** | ~0.001元/千token | ~0.03元/千token | ~0.02元/千token | ~0.01元/千token |
| **接口兼容** | OpenAI 兼容 | 原生 | Anthropic SDK | OpenAI 兼容 |
| **代码生成** | 优秀 | 优秀 | 优秀 | 良好 |
| **中文理解** | 优秀 | 良好 | 良好 | 优秀 |

**选择 DeepSeek-V3 的理由：**
1. 成本优势：价格约为 GPT-4o 的 1/30，学生项目经济可行
2. 接口兼容：OpenAI 兼容格式，通过 LangChain `ChatOpenAI` 封装调用，切换模型只需改环境变量
3. 代码生成能力强：在 HumanEval 等基准上表现接近 GPT-4
4. 中文理解优秀：项目 prompt 和评测样例以中文为主

**模型切换方案：** `experiments/config.yaml` 中 `model.model_name` 可配置，切换到 GPT-4o 只需修改配置文件，无需改代码。

## 5. 决策总结

| 选型项 | 选择 | 理由 |
|--------|------|------|
| Agent 框架 | LangGraph 0.2.x | 唯一满足条件路由+状态管理+路由可编程三项硬性条件 |
| LLM 模型 | DeepSeek-V3 | 成本优势 + OpenAI 兼容 + 中文理解优秀 |
| Web 框架 | Flask | 轻量级，3个 API 路由够用，无需 FastAPI 的异步特性 |
| 状态类型 | Pydantic BaseModel | 运行时类型校验 + JSON 序列化 + LangGraph 原生支持 |
| Token 计数 | tiktoken | 与 API 计费口径一致，未安装时回退字符数估算 |
| 配置管理 | YAML + 环境变量 | 优先级：环境变量 > YAML > 硬编码默认值 |
| 测试框架 | pytest | Python 生态标准选择 |
| CI/CD | GitHub Actions | 与 GitHub 仓库原生集成，免费额度足够 |
